LANGCHAIN NOTES FOR RAG CHATBOT
================================

Purpose of this file
--------------------
This file is designed to be placed inside a RAG chatbot knowledge folder such as:

data/notes/langchain_notes_for_rag.txt

It explains LangChain concepts in simple language so a RAG chatbot can retrieve stronger context when users ask about LangChain, RAG, retrievers, chains, prompts, chat history, and Streamlit integration.

------------------------------------------------------------
1. What is LangChain?
------------------------------------------------------------

LangChain is a framework for building applications with language models. It helps developers connect LLMs with prompts, documents, tools, retrievers, memory, APIs, and application logic.

In a basic LLM app, the flow is:

User input
-> Prompt
-> LLM
-> Output

In a RAG app, the flow becomes:

User input
-> Retriever searches external documents
-> Retrieved context is added to the prompt
-> LLM answers using the context
-> Output is shown to the user

LangChain is useful because it provides standard building blocks for these steps.

Main LangChain building blocks:
- Documents
- Document loaders
- Text splitters
- Embedding models
- Vector stores
- Retrievers
- Prompt templates
- Chat models
- Output parsers
- Runnables and LCEL chains
- Message history and memory

------------------------------------------------------------
2. What is a Document in LangChain?
------------------------------------------------------------

A Document is a standard LangChain object that stores text and metadata.

Example:

from langchain_core.documents import Document

doc = Document(
    page_content="RAG means Retrieval-Augmented Generation.",
    metadata={"source": "rag_notes.txt", "topic": "RAG"}
)

page_content contains the actual text.
metadata contains extra information about the text, such as source file, page number, topic, or URL.

Documents are important because RAG systems need a consistent format for storing and retrieving knowledge.

In a RAG chatbot:
- The notes become Documents.
- Documents are split into chunks.
- Chunks are embedded.
- Chunks are stored in a vector store.
- The retriever returns relevant Document chunks.

------------------------------------------------------------
3. What is data ingestion?
------------------------------------------------------------

Data ingestion means loading data from a source into the application.

In your chatbot project, data ingestion means reading files from:

data/notes/

Example sources:
- .txt notes
- PDFs
- webpages
- markdown files
- CSV files
- documentation files

Simple ingestion example for .txt files:

from pathlib import Path
from langchain_core.documents import Document

def load_text_documents(folder_path: str):
    docs = []
    folder = Path(folder_path)

    for file_path in folder.glob("*.txt"):
        text = file_path.read_text(encoding="utf-8")
        docs.append(
            Document(
                page_content=text,
                metadata={"source": file_path.name}
            )
        )

    return docs

This function:
1. Opens each .txt file.
2. Reads the text.
3. Converts the text into a LangChain Document.
4. Stores the file name in metadata.

------------------------------------------------------------
4. What is text transformation or text splitting?
------------------------------------------------------------

Text transformation means preparing documents before retrieval. The most common transformation is text splitting.

Large documents are too big to retrieve and send to the LLM directly. So they are split into smaller chunks.

Example:

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=700,
    chunk_overlap=100
)

chunks = splitter.split_documents(documents)

chunk_size controls the approximate size of each chunk.
chunk_overlap repeats some text between neighboring chunks.

Why chunking matters:
- If chunks are too small, they may lose context.
- If chunks are too large, they may include too many unrelated ideas.
- Overlap helps preserve meaning across chunk boundaries.

For beginner RAG:
- chunk_size around 500 to 1000 characters is a good start.
- chunk_overlap around 50 to 150 characters is a good start.

Important difference:

splitter.create_documents(texts)
- Use when you have plain strings.

splitter.split_documents(documents)
- Use when you already have LangChain Document objects.

If your ingestion function returns Document objects, use split_documents.

------------------------------------------------------------
5. What are embeddings?
------------------------------------------------------------

Embeddings convert text into numerical vectors.

Example idea:

"What is RAG?" -> [0.12, -0.44, 0.87, ...]

Similar meanings should have similar vectors.

In RAG:
- Each document chunk is converted into an embedding.
- The user question is also converted into an embedding.
- The vector store compares the question vector with document vectors.
- The most similar chunks are returned.

The embedding model does not generate answers. It only helps search by meaning.

Example:

from langchain_huggingface import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-MiniLM-L6-v2"
)

This model is lightweight and useful for local semantic search.

------------------------------------------------------------
6. What is a vector store?
------------------------------------------------------------

A vector store is a database that stores embeddings and supports similarity search.

Examples:
- Chroma
- FAISS
- Pinecone
- Weaviate
- Qdrant

In your project, Chroma can be used like this:

from langchain_chroma import Chroma

vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="vectorstore/chroma_db"
)

This does:
1. Embed each chunk.
2. Store the vector representation.
3. Save the index in a local folder.

The vector store can perform semantic search:

docs = vectorstore.similarity_search("What is a retriever?", k=3)

This returns the top 3 semantically similar document chunks.

------------------------------------------------------------
7. What is a retriever?
------------------------------------------------------------

A retriever is an interface that returns relevant documents for a query.

A vector store stores and searches data.
A retriever is a chain-friendly wrapper around search.

Example:

retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 3}
)

docs = retriever.invoke("What is RAG?")

The retriever returns a list of Document objects.

Simple analogy:
- Vector store = library database
- Retriever = librarian
- LLM = teacher who explains using the retrieved pages

A retriever is not a copy of the vector store. It is an interface that knows how to fetch relevant documents from the vector store.

------------------------------------------------------------
8. What is k in a retriever?
------------------------------------------------------------

k controls how many document chunks the retriever returns.

Example:

search_kwargs={"k": 3}

means:
Return the top 3 most relevant chunks.

If k is too small:
- The LLM may not get enough information.

If k is too large:
- The LLM may receive noisy or irrelevant context.

Good starting values:
- k=3 for small notes
- k=4 or k=5 for larger notes

------------------------------------------------------------
9. What is a prompt template?
------------------------------------------------------------

A prompt template is a reusable prompt with variables.

Example:

from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "You are a helpful AI/NLP tutor. Use the context to answer."),
        ("human", "Question: {question}\n\nContext:\n{context}")
    ]
)

Variables:
- {question} is replaced with the user question.
- {context} is replaced with retrieved documents.

Prompt templates are useful because they keep prompts organized and reusable.

------------------------------------------------------------
10. SystemMessage, HumanMessage, AIMessage
------------------------------------------------------------

LangChain chat models use messages.

SystemMessage:
- Defines the assistant's behavior or rules.
- Example: "You are a helpful AI tutor."

HumanMessage:
- Represents the user's message.
- Example: "What is RAG?"

AIMessage:
- Represents the assistant's previous response.
- Example: "RAG means Retrieval-Augmented Generation."

In chat history, messages usually follow this pattern:

HumanMessage -> AIMessage -> HumanMessage -> AIMessage

System messages are not normal conversation history. They are instructions.

------------------------------------------------------------
11. What is MessagesPlaceholder?
------------------------------------------------------------

MessagesPlaceholder creates a place in the prompt where chat history can be inserted.

Example:

from langchain_core.prompts import MessagesPlaceholder

prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "You are a helpful assistant."),
        MessagesPlaceholder(variable_name="chat_history"),
        ("human", "{question}")
    ]
)

If chat_history contains previous HumanMessage and AIMessage objects, they are inserted into the prompt.

This helps the model understand follow-up questions.

Example:
User: What is RAG?
Assistant: RAG means Retrieval-Augmented Generation.
User: Explain it with an example.

The model understands that "it" refers to RAG because chat history is included.

------------------------------------------------------------
12. What is LCEL?
------------------------------------------------------------

LCEL means LangChain Expression Language.

It is a way to connect components using the pipe operator:

chain = prompt | llm | parser

This means:
1. Fill the prompt.
2. Send it to the LLM.
3. Parse the output.

The pipe operator sends the output of one component into the next component.

Example:

from langchain_core.output_parsers import StrOutputParser

chain = prompt | llm | StrOutputParser()

response = chain.invoke({"question": "What is RAG?", "context": "RAG means..."})

LCEL makes pipelines readable and composable.

------------------------------------------------------------
13. What is RunnablePassthrough?
------------------------------------------------------------

RunnablePassthrough passes the input forward unchanged.

It is useful when you want to keep the original input while adding new fields.

Example:

from langchain_core.runnables import RunnablePassthrough

chain = (
    RunnablePassthrough.assign(context=retrieve_context)
    | prompt
    | llm
    | StrOutputParser()
)

If the input is:

{
  "question": "What is RAG?",
  "chat_history": []
}

then RunnablePassthrough.assign(context=retrieve_context) produces:

{
  "question": "What is RAG?",
  "chat_history": [],
  "context": "retrieved notes about RAG..."
}

So it keeps the original input and adds a new key called context.

This is a clear and solid pattern for RAG chains.

------------------------------------------------------------
14. Recommended solid RAG chain pattern
------------------------------------------------------------

Use this pattern as your home-base chain:

def retrieve_context(inputs):
    question = inputs["question"]
    docs = retriever.invoke(question)
    context = format_docs(docs)
    return context

rag_chain = (
    RunnablePassthrough.assign(context=retrieve_context)
    | prompt
    | llm
    | StrOutputParser()
)

Input:
{
  "question": user_input,
  "chat_history": chat_history
}

After assign:
{
  "question": user_input,
  "chat_history": chat_history,
  "context": retrieved_context
}

Then the prompt receives:
- question
- chat_history
- context

Then the LLM generates an answer.

Mental model:

input dictionary
-> add context
-> prompt
-> LLM
-> output parser

15. Full rag_chain.py example
------------------------------------------------------------

import os
from langchain_groq import ChatGroq
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough


def format_docs(docs):
    return "\n\n".join(
        f"Source: {doc.metadata.get('source', 'unknown')}\n{doc.page_content}"
        for doc in docs
    )


def create_rag_chain(vectorstore):
    retriever = vectorstore.as_retriever(
        search_type="similarity",
        search_kwargs={"k": 3}
    )

    llm = ChatGroq(
        groq_api_key=os.getenv("GROQ_API_KEY"),
        model="llama-3.1-8b-instant"
    )

    prompt = ChatPromptTemplate.from_messages(
        [
            (
                "system",
                '''
You are an AI/NLP learning assistant.

Answer the user's question using the retrieved context.
Use simple language, examples, and step-by-step explanation.
If the context does not contain the answer, say that the notes do not contain enough information.

Context:
{context}
'''
            ),
            MessagesPlaceholder(variable_name="chat_history"),
            ("human", "{question}")
        ]
    )

    def retrieve_context(inputs):
        docs = retriever.invoke(inputs["question"])
        return format_docs(docs)

    rag_chain = (
        RunnablePassthrough.assign(context=retrieve_context)
        | prompt
        | llm
        | StrOutputParser()
    )

    return rag_chain, retriever

------------------------------------------------------------
16. What is StrOutputParser?
------------------------------------------------------------

LLMs often return AIMessage objects.

Example:

AIMessage(content="RAG means Retrieval-Augmented Generation.")

StrOutputParser converts the model output into a plain string.

Example:

chain = prompt | llm | StrOutputParser()

Now the output is:

"RAG means Retrieval-Augmented Generation."

This is useful for Streamlit because st.markdown(answer) expects displayable text.

------------------------------------------------------------
17. What is chat history?
------------------------------------------------------------

Chat history stores previous user and assistant messages.

In Streamlit, you may keep two histories:

1. UI history:
st.session_state.messages

Example:
[
  {"role": "user", "content": "What is RAG?"},
  {"role": "assistant", "content": "RAG means..."}
]

This is for displaying chat bubbles.

2. LangChain model history:
st.session_state.chat_history

Example:
[
  HumanMessage(content="What is RAG?"),
  AIMessage(content="RAG means...")
]

This is for passing previous messages to the LLM.

UI history is for screen display.
LangChain chat history is for model context.

------------------------------------------------------------
18. What is RunnableWithMessageHistory?
------------------------------------------------------------

RunnableWithMessageHistory automatically manages chat history for a chain.

Without it:
- You manually pass chat_history every time.
- You manually append HumanMessage and AIMessage.

With it:
- LangChain loads history using session_id.
- LangChain adds the new user message.
- LangChain saves the assistant reply.

Example concept:

store = {}

def get_session_history(session_id):
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

with_message_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="question",
    history_messages_key="chat_history"
)

Manual chat_history is easier for beginners.
RunnableWithMessageHistory is better when you want automatic session-based memory.

------------------------------------------------------------
19. Why trim chat history?
------------------------------------------------------------

If chat history grows forever, the input can exceed the LLM context window.

Trimming means keeping only the most useful recent messages.

Simple version:

recent_history = st.session_state.chat_history[-8:]

This keeps the last 8 messages.

Advanced version uses trim_messages to keep messages under a token limit.

Reason:
- Faster response
- Lower cost
- Avoid context overflow
- Keep recent relevant conversation

------------------------------------------------------------
20. What is history-aware retrieval?
------------------------------------------------------------

Basic RAG retrieves using only the latest question.

Problem:
User asks: "What is RAG?"
Then asks: "Explain it with example."

The retriever searches "Explain it with example", which may be weak because "it" is unclear.

History-aware retrieval first rewrites the question:

"Explain it with example"
-> "Explain RAG with an example"

Then it retrieves documents using the rewritten standalone question.

This improves conversational RAG.

Conceptual flow:
chat_history + latest question
-> rewrite into standalone question
-> retrieve documents
-> answer with context

------------------------------------------------------------
21. What is create_history_aware_retriever?
------------------------------------------------------------

create_history_aware_retriever creates a retriever that can use chat history to reformulate follow-up questions.

It needs:
- LLM
- retriever
- contextualization prompt

The contextualization prompt should not include {context}.
It only needs:
- chat_history
- input

Example system prompt:
"Given chat history and the latest user question, rewrite the question as a standalone question. Do not answer it."

Important:
Question rewriting prompt:
- Uses input + chat_history
- Does not use context

Answering prompt:
- Uses input + chat_history + context

If the rewriting prompt accidentally includes {context}, LangChain may raise a missing variable error.

------------------------------------------------------------
22. What is a RAG prompt?
------------------------------------------------------------

A RAG prompt usually contains:
- assistant role
- retrieved context
- user question
- instruction about what to do if context is insufficient

Example:

You are an AI/NLP learning assistant.
Use the retrieved context to answer the question.
If the answer is not in the context, say the notes do not contain enough information.

Context:
{context}

Question:
{question}

Good RAG prompts prevent hallucination by telling the LLM to stay grounded in the retrieved context.

------------------------------------------------------------
23. What is hallucination in RAG?
------------------------------------------------------------

Hallucination means the model produces information that is not supported by the retrieved context.

RAG reduces hallucination by giving the model relevant documents.

But RAG does not completely remove hallucination.

To reduce hallucination:
- Use clean notes
- Use good chunking
- Use a good embedding model
- Use an appropriate k value
- Ask the LLM to answer only from context
- Show sources
- Evaluate retrieval quality

This makes the chatbot more transparent and professional.

For CV, this shows:

27. Common LangChain errors and meanings
------------------------------------------------------------

Error:
ModuleNotFoundError: No module named 'langchain.chains'

Meaning:
The course uses an older LangChain import path.
Newer versions may require langchain_classic or LCEL instead.

Better:
Use LCEL chains manually:
RunnablePassthrough.assign(...) | prompt | llm | StrOutputParser()

Error:
TypeError: expected string or bytes-like object

Meaning:
You used create_documents on Document objects.
Use split_documents instead.

Wrong:
chunks = splitter.create_documents(documents)

Correct:
chunks = splitter.split_documents(documents)

Error:
Chroma.__init__() got an unexpected keyword argument 'persist_dictionary'

Meaning:
Spelling mistake.

Wrong:
persist_dictionary

Correct:
persist_directory

Error:
Input to ChatPromptTemplate is missing variables {'context'}

Meaning:
Your prompt contains {context}, but your chain did not provide context.
Make sure the RAG chain adds context before the prompt.

Error:
NameError: name 'file_path' is not defined

Meaning:
Variable name mismatch or indentation issue.
Make sure file_path is defined in the loop:
for file_path in folder.glob("*.txt"):

------------------------------------------------------------

-> prepare variables
-> prompt
-> model
-> parser/output

For RAG:

For conversational RAG:

30. Manual version of the same RAG chain
------------------------------------------------------------

def answer_question(question, chat_history):
    docs = retriever.invoke(question)
    context = format_docs(docs)

    final_prompt = prompt.invoke(
        {
            "question": question,
            "chat_history": chat_history,
            "context": context
        }
    )

    response = llm.invoke(final_prompt)
    answer = StrOutputParser().invoke(response)
    return answer

This manual version does the same thing as the LCEL chain.

LCEL is just a shorter and more composable way to write it.

31. LangChain Q&A examples
------------------------------------------------------------

Question: What is LangChain?
Answer: LangChain is a framework for building applications with language models. It provides components for prompts, chains, retrievers, memory, tools, and document-based workflows.

Question: What is LCEL?
Answer: LCEL means LangChain Expression Language. It lets developers connect components using the pipe operator, such as prompt | model | parser.

Question: What is a prompt template?
Answer: A prompt template is a reusable prompt with variables. The variables are filled at runtime with user input, context, or chat history.

Question: What is a retriever?
Answer: A retriever is an interface that returns relevant documents for a query. In RAG, it often searches a vector store.

Question: What is a vector store?
Answer: A vector store stores embeddings of document chunks and supports semantic similarity search.

Question: What is a RAG chain?
Answer: A RAG chain retrieves relevant context from a knowledge base and sends that context with the user's question to the LLM.

Question: What is RunnablePassthrough?
Answer: RunnablePassthrough passes the input forward unchanged. It can also add new fields using assign.

Question: What is StrOutputParser?
Answer: StrOutputParser converts model outputs such as AIMessage objects into plain text strings.

Question: Why use MessagesPlaceholder?
Answer: MessagesPlaceholder inserts previous chat messages into a prompt, helping the model understand conversation history.

Question: Why does my chatbot need chat history?
Answer: Chat history helps the model understand follow-up questions and previous context in a conversation.

Question: Does chat history automatically improve retrieval?
Answer: Not always. Basic RAG retrieves using the latest question only. History-aware retrieval is needed to rewrite follow-up questions before retrieval.

Question: What is the difference between manual chat_history and RunnableWithMessageHistory?
Answer: Manual chat_history means you append and pass messages yourself. RunnableWithMessageHistory lets LangChain manage session-based history automatically.

Question: Why should I show retrieved sources?
Answer: Showing retrieved sources makes the chatbot more transparent and helps users trust the answer. It also helps debug retrieval quality.

Question: What is the difference between RAG and fine-tuning?
Answer: RAG retrieves external knowledge at inference time. Fine-tuning changes model parameters using training data. RAG is better for updating knowledge without retraining.

Question: Can DistilBERT generate chatbot answers?
Answer: DistilBERT is mainly for understanding tasks like classification, not long text generation. In a chatbot, it can be used for intent classification while an LLM generates the answer.
